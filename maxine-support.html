HFA's technical and operational read on the Addendum
Section-by-section response to Sarita's Handbook v1.1. This document is HFA's formal input before the main Handbook is updated.
The system principle holds — HFA fully endorses it
"Content engine first. Automation second. Conductor approval always."
Every HFA workflow in the Maxine system is built around this. Nothing publishes without Sarita's approval gate. The automation moves content from Maxine to Notion and from Notion to platforms — it does not decide, select, or approve anything.
Section 1 — Content Engine Library Doctrine
HFA read: Aligned. Notion stores and retrieves. n8n connects Maxine's output to Notion and connects Notion's approved assets to publishing destinations. The library never becomes the campaign brain — Maxine owns that. HFA's workflows will write to and read from these libraries with full field discipline, using the exact field structure Sarita defines.
Section 2 — Domain-Specific Library Pattern
HFA read: Clean architecture. Two libraries is the right call — the fields differ enough between music campaign and traditional campaign work that a single library would become unmanageable. HFA will build both Notion databases with the exact fields from the Addendum, plus add a Campaign Mode field that maps to Maxine's 11 modes. This is the link between Section 2 and Section 6.
Section 3 — Publishing Queue Decision Rule
HFA recommendation: Do not build the queue yet. The content library publishing fields handle single-platform, manual, or simple scheduled posts. Build the Publishing Queue as a separate Supabase operations table only when one asset needs to go to multiple platforms on different dates with different versions. Trigger condition: when the Lowcountry Jazz Festival campaign requires Facebook + Instagram + Email simultaneously with platform-specific versions. Until then — library fields are sufficient.
Section 4 — JSON Routing Contract Rule
HFA read: This is HFA's direct implementation territory. n8n reads the content_domain field from Maxine's output JSON and routes to the correct Notion library. HFA proposes extending the JSON beyond the two-field example in the Addendum to include campaign mode, status, platform targets, and approval flag. See Tab 04 for the full JSON schema proposal. The rule that JSON records but does not decide is core to everything HFA builds here — the cognition boundary holds.
Section 5 — Joint Client Orchestration
HFA read: All four TBDs can be answered now. See Tab 05 for the full ownership JSON with HFA's positions on each. Short version: client relationship owner is determined at intake by who originated the relationship. Database owner is split — Notion structure is AI Aunties' domain, Supabase operations is HFA's. Publishing operator is HFA for automated multi-platform delivery, Conductor for manual single posts. Orchestration owner is HFA as Mode 1 (Auntie Workflow Extension), with Conductor approval always as the non-negotiable gate.
Section 6 — Campaign Mode Routing
HFA can solve this in n8n. Before Maxine runs, an intake workflow reads the brief, identifies intent signals, and presents the correct mode options when intent is ambiguous — surfacing the exact question Sarita wrote: "Do you want a quick ticket-push post, a deeper cultural story post, or a story-led ticket movement post?" This is a pre-Auntie routing workflow (Category 1 in the scope architecture) that HFA builds as part of the intake bundle. See Workflow 01 in Tab 02.
Section 7 — Evidence to Preserve from Testing
HFA recommendation: The rejection patterns (AI-ism drift, campaign-copy patterns) should be logged to Supabase — not just stored in Notion. Supabase allows queryable pattern analysis over time. A weekly summary workflow pulls the most common rejection patterns and sends them to Sarita as a brief, enabling systematic improvement of Maxine's taste pack. See Workflow 05 in Tab 02.
The one thing HFA wants to flag before Sarita updates the main Handbook
The Addendum defines Campaign Mode as a product requirement for commercial deployment (Section 6). This needs to be a Notion field in both libraries — so every output is tagged with its mode. This allows Sarita to filter the library by mode, build pattern reports, and eventually build a mode-specific performance view. HFA is proposing this as a required field, not optional.

HFA's answers to the Section 5 TBDs
These are HFA's definitive operational positions. Sarita should review these before updating the main Commercial Handbook. Any position marked ⚠️ should be discussed before it is codified.
client_relationship_owner
Was: TBD
HFA answer: The brand who originated the client relationship. Determined at engagement intake and recorded in the engagement scope document. If AI Aunties sourced the client (festival, cultural org, NPO, founder), AI Aunties owns the client relationship. If HFA sourced the client (solopreneur, small business, workflow subscription), HFA owns it. For joint engagements where both brands are co-named, the lead varies by what the client primarily bought.
Implementation: add "client_relationship_owner" field to the engagement intake form. Value locked at deal start. Cannot be changed without both founders agreeing.
database_owner
Was: TBD
HFA answer: Split ownership. Notion field schema and library structure is governed by AI Aunties — Sarita defines what fields exist and what they mean. HFA builds the n8n connections that write to and read from those databases. Supabase is HFA's domain — publishing operations, rejection log, session data, model registry, and performance tracking live in Supabase, built and owned by HFA. Neither party modifies the other's database layer without discussion.
publishing_operator
Was: Human First Automations or client, depending on scope
HFA answer: Two clear tiers.

Tier 1 — Manual: Single platform, Conductor posts directly. HFA workflow sends a notification with copy-ready formatted content. No automation beyond the notification.

Tier 2 — Automated (Phase 2): Multi-platform, scheduled, with version adaptation. HFA builds and operates the publishing pipeline. Conductor approval remains the hard gate — automation only fires after "Ready to Publish" is TRUE.

The Addendum's decision rule (library publishing fields now, separate queue when volume requires) stays intact.
orchestration_owner
Was: TBD
HFA answer: Human First Automations — Mode 1 (Auntie Workflow Extension). HFA owns the orchestration of all n8n workflows in this system: intake routing, output routing, library filing, approval gating, publishing mechanics, rejection logging, and calendar sequencing. This is governed by the AI Aunties Commercial Build & Deployment Handbook. AI Aunties owns the Conductor role — the decision point that every automated workflow must pause at before any content moves to publication.
⚠️ Discussion point: For joint client engagements where HFA has an independent relationship with the client, the orchestration boundary should be clarified in writing. HFA Mode 2 (downstream) vs. Mode 1 (inside AIA scope) may overlap on some joint engagements. This should be addressed before first joint commercial Maxine deployment.
